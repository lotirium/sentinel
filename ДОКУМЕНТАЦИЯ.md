# CodeSentinel — Документация проекта

---

## Идея и контекст

Проект создан как **use case для OpenClaw** — демонстрация того, на что способен OpenClaw-агент в реальном production-сценарии.

**OpenClaw** — это ИИ-агент (на базе Claude от Anthropic), который умеет работать с кодом: читать файлы, понимать контекст, вносить изменения, запускать команды и верифицировать результат. Мы взяли эту возможность и встроили её в полноценный инцидентный пайплайн.

---

## Что такое CodeSentinel?

**CodeSentinel** — это система автономного реагирования на production-инциденты, построенная вокруг OpenClaw.

Когда в production-сервисе появляется баг и падают тесты — **не нужен инженер**. CodeSentinel:
1. Сам обнаруживает проблему
2. Сам вызывает OpenClaw
3. OpenClaw читает код, ставит диагноз, пишет патч, прогоняет тесты
4. Система верифицирует исправление
5. Отправляет отчёт в Telegram

Всё это происходит автономно, за секунды, с полной прозрачностью через веб-дашборд.

---

## OpenClaw — сердце системы

### Что такое OpenClaw и как он используется здесь

OpenClaw — CLI-инструмент, запускающий ИИ-агента в контексте проекта. В этом проекте он выступает как **автономный разработчик-дежурный**.

Вызов OpenClaw из кода:
```bash
openclaw agent --agent main --message "$(cat prompt_file.txt)"
```

Система передаёт OpenClaw структурированный промпт с точным заданием, OpenClaw выполняет его как настоящий разработчик: читает файлы, пишет код, запускает тесты.

---

### Сценарий 1: Автономное исправление бага

Это основной сценарий. Когда тесты упали, система формирует промпт для OpenClaw, например:

```
The file is at /mnt/c/Users/Asus/Desktop/sentinel/main.py.
Fix the ZeroDivisionError: when request.features is empty,
return PredictionResponse(prediction=0.0, feature_count=0)
before touching the tensor.
Run: pytest tests/test_api.py
Commit with message: 'fix: handle empty input in predict'

Traceback:
prediction = float(tensor.sum()) / len(request.features)
ZeroDivisionError: float division by zero
```

OpenClaw получает это задание и выполняет его шаг за шагом:
- Открывает `main.py`
- Находит строку с ошибкой
- Добавляет guard-проверку перед делением
- Запускает `pytest tests/test_api.py`
- Делает git commit с нужным сообщением
- Возвращает результат

**Важно:** если несколько сервисов сломаны одновременно — система запускает несколько экземпляров OpenClaw **параллельно** через `asyncio.gather`. Время восстановления не складывается, а остаётся равным времени самого долгого агента.

---

### Сценарий 2: Редактирование кода через Telegram

Второй use case — OpenClaw как **персональный разработчик в Telegram**. Пользователь пишет в чат обычным языком:

```
add a /metrics endpoint to main.py
change the prediction formula to use the median
add input validation for negative numbers
```

Система передаёт это сообщение OpenClaw вместе с контекстом проекта. OpenClaw вносит изменение, запускает тесты, подтверждает результат. Прогресс стримится в чат в реальном времени — пользователь видит каждый шаг агента по мере выполнения.

### Фаза 0: Инжекция бага

Для демонстрации система **намеренно вносит баг** в код сервиса. Это позволяет показать полный цикл работы "вживую". В реальном применении эту фазу заменяет настоящий баг из production.

**Три подготовленных бага:**

| Сервис | Баг | Тип ошибки |
|--------|-----|------------|
| `main.py` | Деление на длину пустого списка | `ZeroDivisionError` |
| `services/torch_worker.py` | Тензор создан с `dtype=int32` вместо `float32` | `RuntimeError` |
| `services/data_pipeline.py` | Доступ к ключу словаря через `[]` вместо `.get()` | `KeyError` |

---

### Фаза 1: Запуск тестов

Система запускает `pytest` для каждого сервиса и фиксирует, кто упал.

```

```

### Фаза 2: Алерт

Telegram получает сообщение об инциденте:

```
Status: 🤖 Engaging CodeSentinel AI...
```

---

### Фаза 3: OpenClaw исправляет

Для каждого упавшего сервиса:

1. На дашборде отображается **цепочка рассуждений агента** (reasoning chain):
   - Аномалия обнаружена
   - Гипотеза 1: несовместимость версий PyTorch → отклонена
   - Гипотеза 2: необработанный edge case → подтверждена
   - Стратегия: добавить guard перед делением
   - Патч применяется

2. OpenClaw реально вносит изменение в файл

3. Проводится **security scan** патча:
   - Нет захардкоженных секретов
   - Нет SQL-инъекций через дефолтные значения
   - Нет новых сетевых портов
   - Нет утечки внутренних путей
   - CVE-скан: новые зависимости не добавлены

---

### Фаза 4: Верификация

Тесты запускаются повторно. Если `pytest` возвращает код 0 — патч принят. Если нет — система сообщает о необходимости ручного вмешательства.

---

### Фаза 5: Закрытие инцидента

```
✅ CLUSTER RESTORED

Services remediated: price-prediction-api, torch-inference-worker
All tests green. Changes committed.
CodeSentinel AI — incident resolved autonomously.
```

## Сервисы, которые мониторит система

### Price Prediction API (`main.py`)

FastAPI-сервис предсказания цены на основе числовых признаков.

| Эндпоинт | Метод | Описание |
|----------|-------|----------|
| `/health` | GET | Проверка работоспособности |
| `/predict` | POST | Предсказание по списку признаков |

```json
// Запрос
{ "features": [10.0, 20.0, 30.0] }

// Ответ
{ "prediction": 20.0, "feature_count": 3 }
```

**Бага в нём:** при пустом списке `features` — деление на ноль. OpenClaw исправляет это, добавляя проверку до вычислений.

---

### Torch Inference Worker (`services/torch_worker.py`)

Сервис вычисления схожести между векторами через матричное умножение PyTorch.

**Бага:** тензор ключей создаётся с `dtype=torch.int32`, тогда как `torch.mm()` требует `float`. OpenClaw меняет тип на `float32`.

---

### Data Pipeline (`services/data_pipeline.py`)

Сервис пакетной обработки записей с ценами.

**Бага:** прямой доступ `record["price"]` бросает `KeyError` на записях без поля `price`. OpenClaw заменяет на `record.get("price", 0.0)`.

## Веб-дашборд

Запускается на порту **8080**. Показывает весь инцидентный цикл в реальном времени.

**Технология стриминга: SSE (Server-Sent Events)**

Один запуск симуляции создаёт один поток событий. Все подключённые клиенты — браузер и Telegram-бот — получают **одни и те же события** через broadcast-очередь. Симуляция никогда не запускается дважды.

**Живой `/predict` эндпоинт на дашборде:**
Каждый раз перезагружает `main.py` с диска через `importlib`. Это значит: если баг ещё не исправлен — вернёт реальный `ZeroDivisionError 500`. Если OpenClaw уже применил патч — вернёт корректный ответ `200`. Никакой симуляции — реальный код, реальный результат.

## Telegram-бот

Запускается автоматически вместе с дашбордом в фоновом потоке.

| Команда | Что делает |
|---------|-----------|
| `/help` | Показывает справку |
| `/status` | Текущее состояние системы |
| `Activate the test with simulation` | Запускает полный инцидентный цикл |
| Любой текст | Передаётся как задание в OpenClaw |

**Как работает стриминг ответа от OpenClaw в Telegram:**

Пока OpenClaw работает — бот показывает индикатор "печатает...". Строки с ключевыми словами (`editing`, `patching`, `running`, `passed`, `failed` и др.) сразу отправляются в чат. По завершении — финальные 6 строк вывода как итог.

## Ключевая ценность проекта

**Без CodeSentinel:** упал сервис → дежурный инженер получает алерт → просыпается → читает логи → ищет причину → пишет патч → деплоит → проверяет. 30–60 минут простоя.

**С CodeSentinel + OpenClaw:** упал сервис → система обнаруживает → OpenClaw анализирует, чинит, верифицирует → Telegram: "инцидент закрыт". 2–3 минуты. Инженер даже не просыпается.

Это **реальный use case OpenClaw**: агент встроен в production-пайплайн как автономный исполнитель, а не как чат-ассистент.

# Что показывать на демо — короткий ответ

**Вопрос:** Что будете показывать сегодня на демо?

**Ответ:**

Мы показываем **CodeSentinel** — систему автономного реагирования на production-инциденты, построенную на основе **OpenClaw**.

Это наш use case для OpenClaw: вместо того чтобы использовать агента как чат-ассистента, мы встроили его в полноценный DevOps-пайплайн.

На демо происходит следующее: в реальный FastAPI-сервис вносится баг — деление на ноль при пустом входе. Система запускает тесты, они падают. Автоматически уходит тревога в Telegram. Затем вызывается OpenClaw: агент сам читает код, формулирует гипотезы о причине, выбирает стратегию исправления, пишет патч, прогоняет тесты повторно и делает коммит. Всё это видно в реальном времени на веб-дашборде — каждый шаг рассуждений агента, security scan патча, метрики сервисов. В финале Telegram получает подтверждение: инцидент закрыт, все тесты зелёные.

**Главная идея: OpenClaw как дежурный инженер — работает 24/7, реагирует за минуты, объясняет каждое решение.**
